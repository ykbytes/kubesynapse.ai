What If Agents Were
Kubernetes Resources?

apiVersion: kubesynapse.ai/v1alpha1
kind: AIAgent
metadata:
  name: security-analyst
  namespace: production
spec:
  runtime: opencode
  model: gpt-4o
  systemPrompt: |
    Analyze security alerts...
  mcpSidecars:
    - code-exec
    - web-search
  policy: security-guardrails

Operator Reconciles Into:

  • StatefulSet security-analyst-runtime
  • Service security-analyst-svc
  • PVC security-analyst-workspace
  • NetworkPolicy security-analyst-netpol
  • MCP Sidecars: code-exec, web-search

One YAML declares the agent. The operator provisions everything. Kubernetes manages the lifecycle.

Tools via MCP

Model Context Protocol — Two Integration Patterns

Per-Agent Sidecars

Localhost communication. Pod-local trust. Each agent gets exactly the tools it needs.

code-exec web-search browser database git github kubernetes messaging RAG documents

10 bundled tool sidecars, plus a separate collector sidecar and shared MCP hub integrations

Shared MCP Hub

McpConnection CRD defines saved integrations. Multiple agents share connections.

kind: McpConnection
metadata:
  name: jira-integration
spec:
  transport: sse
  url: https://mcp.jira.internal
  shared: true

Security Is Layered, Not Bolted On

Who Is This For?

Platform Engineers

Agents don't fit cluster governance
No standard deployment contract
Bypass resource controls
CRDs + Helm + RBAC + NetworkPolicies = agents as first-class cluster citizens

DevOps / SRE

Can't tell what agents are doing
No visibility into failures
No anomaly alerting
Observatory + traces + signal_watch + structured runtime events

Enterprise AI Teams

Need governance and audit trails
Multiple models, changing providers
Workflows need approval gates
AgentPolicy + LiteLLM abstraction + HITL approvals + full audit

Ship AI agents the same way
you ship everything else.

As Kubernetes resources with observable runtime behavior.
